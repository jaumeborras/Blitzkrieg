🧩 Brecha entre repositorio y CMS
El repositorio actual es un DAM (Digital Asset Management) muy bien construido. Lo que lo separa de un CMS son básicamente tres piezas: la capa de publicación (asignar contenido a páginas), la capa de texto (gestionar el copy de la web) y la capa de integración (que la web consuma el repositorio en tiempo real). Las tres son construibles de forma incremental sin interrumpir el servicio actual.

Necesidades a Corto Plazo

5 pilares críticos para transformar el repositorio en un CMS real

🏷️
1. Metadatos orientados a publicación web
Los archivos actuales tienen etiquetas descriptivas (tipo, categoría, resolución). Hay que añadir campos específicos de CMS: título SEO, texto alternativo (alt text), descripción pública y sobre todo a qué páginas o secciones de la web está asignada cada foto. Sin esto, el repositorio sigue siendo un almacén, no un CMS.
Alta prioridad Inmediato
🗺️
2. Gestor de contenido por página/sección
Una vista donde el equipo de marketing pueda ver, por ejemplo, "Página principal → Hero banner" y arrastrar desde la biblioteca la foto que quiera publicar. Este es el salto conceptual clave: pasar de "gestionar archivos" a "gestionar páginas".
Alta prioridad 1-2 meses
🔄
3. Sistema de versiones y publicación
Distinguir entre borradores, contenido en revisión y contenido publicado. Con un flujo de aprobación mínimo (editor → aprueba manager) ya os desvinculáis completamente de terceros para gestionar cambios en la web.
Media prioridad 2-3 meses
✍️
4. Gestión de textos junto a las imágenes
Un CMS real necesita editar también los textos de la web: títulos, descripciones de habitaciones, precios de menú, horarios de piscina... Podría ser tan sencillo como campos de texto asociados a cada "sección" de la web.
Media prioridad 3-5 meses
🔌
5. API o conector con la web actual
Para no hacer una migración brusca, lo más inteligente es que la web consuma el repositorio vía API. Así podéis ir trasladando secciones poco a poco, sin cortar nada en producción. La web actual sigue funcionando mientras preparáis el nuevo sistema en paralelo.
Crítico para migración 2-3 meses
💬 El concepto clave
La diferencia entre un DAM y un CMS es simple: el DAM dice "aquí están las fotos". El CMS dice "esta foto va en este sitio de la web, con este texto, aprobado por esta persona, publicado hoy". El paso del uno al otro es un trabajo de conectar contexto, no de reconstruir desde cero.

Roadmap de Implementación

Haz clic en cada fase para ver los detalles y entregables

🏷️ Fase 1 — Metadatos de publicación
Inmediato
Añadir campo "asignación a sección web" en los metadatos de cada archivo
Entregables:
  • Campo "Sección web destino" en ficha de archivo
  • Campo "Título SEO" y "Alt text" (rellenable y editable)
  • Campo "Estado" (borrador / listo / publicado)
  • Migración de las 18 fotos actuales con los nuevos campos
🗺️ Fase 2 — Mapa de la web
1-2 meses
Construir la vista de "árbol de páginas" con sus slots de contenido
Entregables:
  • Vista árbol con todas las páginas de www.prinsotel.com
  • Slots visuales (Hero, galería, thumbnail...) por sección
  • Drag & drop desde biblioteca a slots
  • Indicador de "este slot está vacío / desactualizado"
🔌 Fase 3 — API + Conector web
2-3 meses
API pública del repositorio + primer conector con la web actual
Entregables:
  • API REST: GET /contenido/{seccion} devuelve imagen + textos
  • Primera sección de prinsotel.com consumiendo la API
  • Caché y CDN para servir imágenes optimizadas
  • Panel de monitorización de peticiones
✍️ Fase 4 — Editor de textos + Aprobaciones
3-5 meses
Editor de textos integrado + flujo de aprobación editor → manager
Entregables:
  • Editor rich text por sección (títulos, cuerpo, CTA)
  • Roles: Editor de contenido / Aprobador / Admin
  • Notificaciones de revisión pendiente
  • Historial de versiones de texto
🚀 Fase 5 — Migración total
6-9 meses
Migración total y desconexión del CMS externo
Entregables:
  • 100% de páginas de prinsotel.com consumiendo el CMS propio
  • Migración del historial de contenido del CMS externo
  • Documentación y formación del equipo
  • Desconexión y baja del CMS externo → ahorro recurrente

Aplica Ya Mismo

6 mejoras que podéis incorporar hoy sin grandes desarrollos

⚡
Estas sugerencias no requieren grandes desarrollos y añaden valor inmediato al repositorio existente. Son el mejor retorno sobre inversión a corto plazo.
🤖
Alt text automático con IA
La misma IA que ya etiqueta las fotos podría generar un borrador de texto alternativo para accesibilidad y SEO. Solo necesitáis añadir ese campo en la ficha del archivo. El equipo lo revisa y aprueba. Impacto inmediato en posicionamiento orgánico.
Disponible hoySEO + Accesibilidad
👁️
Vista previa contextual
Al hacer clic en una foto, mostrar en qué páginas de la web está siendo usada actualmente. Evita publicar accidentalmente una foto que ya está en otro sitio o usar una foto retirada por error.
Disponible hoyPreviene errores
🍂
Sugerencia automática de temporada
Ya tenéis el filtro de temporada, pero si la IA pudiera sugerir automáticamente "esta foto es de verano / Navidad / invierno" basándose en colores, decoración o platos, os ahorraría mucho trabajo manual antes de campañas.
Disponible hoyAhorro manual
⭐
Favoritos o colecciones de trabajo
Que cada usuario pueda crear carpetas privadas de trabajo tipo "fotos para campaña verano 2026" sin mover los archivos del repositorio. Muy útil para que marketing prepare campañas sin tocar la estructura.
Disponible hoyProductividad
📊
Activar estadísticas de uso
La sección "Estadísticas" del menú existe pero está vacía. Registrar qué fotos se publican más, cuáles llevan meses sin usarse, cuáles generan más clics en la web. Datos de rendimiento para tomar decisiones de contenido.
Corto plazoData-driven
🔏
Watermark automático para descarga externa
Si alguna vez se descarga una foto para enviarla a prensa o agencias, que opcionalmente se aplique el logo de Prinsotel de forma automática. Protección de marca sin fricción para el usuario interno.

Visión Final: CMS Propio

El objetivo final a 9 meses vista

🎯
Objetivo: Que cualquier persona del equipo de Prinsotel pueda actualizar cualquier sección de www.prinsotel.com en menos de 3 minutos, sin depender de ningún proveedor externo.
¿Cómo funcionará el CMS final?
1
El equipo sube una foto
La IA la etiqueta automáticamente con tipo, contenido, temporada, alt text sugerido y la coloca en la biblioteca.
2
Se asigna a una sección web
Desde el gestor de páginas, arrastra la foto al slot "Hero — Mallorca Palace" y edita el titular y descripción.
3
El manager revisa y aprueba
Recibe una notificación, revisa la vista previa del cambio en contexto real y aprueba con un clic.
4
La web se actualiza en tiempo real
La API sirve el nuevo contenido. www.prinsotel.com muestra la foto y el texto actualizados. Sin intervención técnica.

Ventaja Competitiva

Por qué el CMS propio de Prinsotel superará a cualquier solución de mercado

🏆 La ventaja irrepetible
La ventaja competitiva real que tenéis sobre cualquier CMS de mercado (Contentful, WordPress, Strapi, etc.) es que el vuestro entiende el contenido hotelero de forma nativa gracias a la IA. Sabe que una imagen es de una piscina exterior en verano, que es ideal para la página de Mallorca Palace, y que tiene orientación horizontal perfecta para un hero banner. Eso es algo que ningún CMS genérico os dará.
CMS genérico de mercado
✗ No entiende el contexto hotelero
✗ Requiere instalación manual de plugins
✗ Sin IA de clasificación de contenido
✗ Licencias recurrentes y vendor lock-in
✗ Agencia necesaria para cambios complejos
✗ DAM y CMS son sistemas separados
CMS propio Prinsotel
✓ IA entrenada en contenido hotelero
✓ Flujos adaptados a operativa hotelera
✓ Etiquetado automático ya en producción
✓ Sin licencias externas a largo plazo
✓ Autonomía total del equipo de marketing
✓ DAM + CMS integrados en un único sistema
🌐 Escalabilidad a otras propiedades
Una vez el CMS esté en producción para Prinsotel, el mismo sistema puede extenderse a cada propiedad del grupo con su propia instancia o sección. La IA puede entrenarse por hotel para aprender las particularidades de contenido de cada uno. El activo tecnológico crece con el grupo hotelero.
